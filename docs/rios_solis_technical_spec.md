## 1. Постановка задачи RPML

**RPML (Repayment Policy for Multiple Loans)** — задача оптимального распределения ежемесячного дохода между несколькими кредитами для минимизации общей суммы выплат.

### Входные данные

| Параметр | Обозначение | Описание |
|----------|-------------|----------|
| `n` | - | Количество кредитов |
| `T` | - | Горизонт планирования (месяцы) |
| `principal[j]` | p_j | Начальная сумма кредита j |
| `interest_rates[j,t]` | i_{j,t} | Месячная процентная ставка кредита j в месяц t |
| `default_rates[j,t]` | h_{j,t} | Штрафная ставка при недоплате (default rate) |
| `min_payment_pct[j]` | α_j | Минимальный платёж как % от баланса |
| `monthly_income[t]` | M_t | Доступный доход в месяц t |
| `release_time[j]` | r_j | Месяц выдачи кредита j (0-indexed) |
| `prepay_penalty[j]` | - | Штраф за досрочное погашение (1e12 = запрещено) |
| `stipulated_amount[j]` | - | Срок кредита в месяцах (НЕ сумма платежа!) |

### Выходные данные

- `X[j,t]` — оптимальный платёж по кредиту j в месяц t
- `objective` — минимальная общая сумма выплат Σ X[j,t]

## 2. MILP формулировка

### 2.1 Переменные решения

| Переменная | Тип | Описание |
|------------|-----|----------|
| `X[j,t]` | ≥ 0 | Платёж по кредиту j в месяц t |
| `B[j,t]` | ≥ 0 | Баланс кредита j в конце месяца t |
| `Z[j,t]` | {0,1} | 1 если кредит j активен в месяц t |
| `S[t]` | ≥ 0 | Накопления (savings) в конце месяца t |
| `C[j,t]` | ≥ 0 | Штраф за недоплату (добавляется к балансу) |

### 2.2 Целевая функция

```
minimize Σ_{j=0}^{n-1} Σ_{t=0}^{T-1} X[j,t]
```

### 2.3 Ограничения

#### Constraint 1: Бюджетное ограничение

```
Σ_j X[j,t] + S[t] ≤ M[t] + S[t-1]    для всех t
```
где `S[-1] = 0` (нет начальных накоплений)

#### Constraint 2: Динамика баланса

**В месяц release (t = r_j):**
```
B[j, r_j] = principal[j] - X[j, r_j]
```
⚠️ **ВАЖНО:** В месяц выдачи кредита проценты НЕ начисляются!

**В последующие месяцы (t > r_j):**
```
B[j,t] = B[j,t-1] * (1 + i[j,t]) - X[j,t] + C[j,t]
```
где `C[j,t]` — штраф за недоплату, добавляемый к балансу.

#### Constraint 3: Неактивность до release

```
Z[j,t] = 0, B[j,t] = 0, X[j,t] = 0    для t < r_j
```

#### Constraint 4: Связь баланса и активности (Big-M)

```
B[j,t] ≤ M * Z[j,t]
X[j,t] ≤ M * Z[j,t]
```
где `M = 1e12` (большое число)

#### Constraint 5: Монотонность активности

Кредит не может "воскреснуть" после погашения:
```
Z[j,t] ≤ Z[j,t-1]    для t > 0
```

#### Constraint 6: Штраф за недоплату (Underpayment penalty)

Если платёж меньше минимального, начисляется штраф:
```
C[j,t] ≥ (α_j * B[j,t-1] - X[j,t]) * (1 + h[j,t]) - M*(1 - Z[j,t])
```

⚠️ **ВАЖНО:** Штраф применяется ТОЛЬКО для `t > r_j`. В месяц release штрафа нет: `C[j, r_j] = 0`

#### Constraint 7: Ограничение платежа сверху

Нельзя заплатить больше, чем текущий долг с процентами:
```
X[j,t] ≤ B[j,t-1] * (1 + i[j,t])    для t > r_j
X[j, r_j] ≤ principal[j]            для t = r_j
```

#### Constraint 8: Полное погашение

```
B[j, T-1] = 0    для всех j
```

## 3. Формат датасета Rios-Solis

### Структура .dat файла

```
Line 1:  n T                           # количество кредитов, горизонт
Line 2:  n_cars n_houses n_cc n_bank   # типы кредитов
Line 3:  principal[0] ... principal[n-1]
Lines 4..(3+n):     interest_rates матрица (n строк × T столбцов)
Lines (4+n)..(3+2n): default_rates матрица (n строк × T столбцов)
Line (4+2n):        min_payment_pct[0] ... min_payment_pct[n-1]
Line (5+2n):        prepay_penalty[0] ... prepay_penalty[n-1]
Line (6+2n):        monthly_income[0] ... monthly_income[T-1]
Line (7+2n):        release_time[0] ... release_time[n-1]
Line (8+2n):        stipulated_amount[0] ... stipulated_amount[n-1]
```

### Специальные значения

- `1e12` (1000000000000.0) — запрещённое значение (prepay_penalty = запрет досрочного погашения)

### Типы кредитов в датасете

| Тип | Характеристики |
|-----|----------------|
| Car loans | Среднесрочные (24-60 мес), средние ставки |
| House loans | Долгосрочные (60-360 мес), низкие ставки, сложнее для solver |
| Credit cards | Краткосрочные (3-24 мес), высокие ставки, высокий min_pct |
| Bank loans | Среднесрочные, различные ставки |

## 4. Baseline алгоритмы

### 4.1 Debt Avalanche (Highest Interest First)

```python
for t in range(T):
    # 1. Рассчитать минимальные платежи для активных кредитов
    min_payments = [α_j * B[j,t-1] for active j]
    
    # 2. Заплатить минимумы
    remaining = M[t] + S[t-1] - sum(min_payments)
    
    # 3. Остаток направить на кредит с максимальной ставкой
    highest_rate_loan = argmax(i[j,t] for active j)
    extra_payment = min(remaining, B[highest_rate_loan] * (1+i))
```

### 4.2 Debt Snowball (Smallest Balance First)

Аналогично Avalanche, но остаток направляется на кредит с **минимальным балансом**.

### 4.3 Сравнение с MILP

| Метрика | Avalanche | Snowball | MILP |
|---------|-----------|----------|------|
| Средняя экономия vs Avalanche | 0% | -2% | **+4.6%** |
| Максимальная экономия | - | - | **+40%** |
| Время решения (8 кредитов) | <1ms | <1ms | ~10 мин |

## 5. Ключевые особенности реализации

### 5.1 Обработка месяца release

```python
# ПРАВИЛЬНО:
if t == r_j:
    B[j,t] = principal[j] - X[j,t]  # БЕЗ процентов
    C[j,t] = 0                       # БЕЗ штрафа
    
# НЕПРАВИЛЬНО (частая ошибка):
if t == r_j:
    B[j,t] = principal[j] * (1+i) - X[j,t]  # ❌ Проценты в первый месяц
```

### 5.2 Бюджетное ограничение в baseline

Если `sum(min_payments) > available_budget`:
```python
# Масштабировать платежи пропорционально
scale = available_budget / sum(min_payments)
actual_payments = [min_pmt * scale for min_pmt in min_payments]
# Применить штраф за недоплату
```

### 5.3 Infeasibility

Модель может быть INFEASIBLE если:
- `sum(monthly_income) < sum(principals)` с учётом процентов
- Горизонт T недостаточен для погашения
- Крупные ипотеки с высокими процентами

## 7. Ограничения модели

1. **Детерминистические доходы** — M[t] известны заранее
2. **Фиксированные ставки** — i[j,t] не зависят от поведения заёмщика
3. **Нет рефинансирования** — нельзя перевести долг между кредитами
4. **Обязательное полное погашение** — B[j,T-1] = 0
5. **Медленное решение** — NP-hard, для 12 кредитов до 30 минут

## 8. Рекомендации для расширения

### Стохастическая версия
- Заменить M[t] на случайную величину M[t] ~ Distribution
- Использовать CVaR вместо минимизации среднего
- Monte Carlo симуляция для оценки рисков

### Ускорение решения
- Декомпозиция по типам кредитов
- Heuristics для начального решения
- Rolling horizon (пересчитывать каждые N месяцев)

### Практические улучшения
- Учёт психологии (Snowball для мелких долгов)
- Интеграция с Open Banking API
- Объяснимость через LLM
